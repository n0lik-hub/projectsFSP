# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## на разработку MVP веб-приложения «VIP Бонус-Арена» (Космический Слалом)

### 1. Введение и цель проекта
Разработать рабочий прототип (MVP) веб-приложения быстрых бонусных игр для VIP-аудитории. Приложение реализует систему игровых комнат с короткими раундами, где участники соревнуются за призовой фонд, используя бонусные баллы.

**Ключевая особенность:** Визуальная оболочка «Космический Слалом» исключает ассоциации с классическим казино. Игровой процесс представлен как гонка космических кораблей, где результат определяется серверной логикой, а клиентская часть (React) отвечает за зрелищную визуализацию и реальное время взаимодействие.

---

### 2. Технологический стек
Для обеспечения производительности, масштабируемости и соответствия современным стандартам выбран следующий стек:

*   **Бэкенд:** Java 17+, Spring Boot 3.x (Web, Data JPA, WebSocket, Security).
*   **Фронтенд:** React 18+ (Functional Components, Hooks, Context API), Vite (сборщик), Axios (HTTP-клиент).
*   **Real-time коммуникация:** WebSocket (STOMP over SockJS) для мгновенного обновления состояния комнат и анимации гонки.
*   **База данных:** PostgreSQL (хранение профилей, истории игр, конфигураций комнат).
*   **Визуализация:** HTML5 Canvas / CSS Animations внутри React-компонентов для отрисовки трассы и кораблей.
*   **Контейнеризация:** Docker & Docker Compose (для унификации запуска на Windows, Linux и macOS).

---

### 3. Функциональные требования

#### 3.1. Пользовательский путь (Frontend на React)
1.  **Главная панель (Dashboard):**
    *   Список активных комнат с фильтрацией (цена входа, размер банка, наличие мест).
    *   Отображение статуса комнаты в реальном времени (обновляется через WebSocket).
    *   Личный кабинет с балансом бонусов и кнопкой быстрого доступа к истории.
2.  **Матчмейкинг и вход:**
    *   Автоматический подбор комнаты по параметрам или ручной выбор.
    *   Модальное окно подтверждения входа с проверкой баланса.
    *   Резервирование баллов при входе (блокировка средств до конца раунда).
3.  **Лобби комнаты:**
    *   Отображение сетки участников (слоты для игроков и ботов).
    *   Таймер обратного отсчета до старта.
    *   Кнопка покупки буста («Импульсный двигатель») с индикацией статуса (куплено/доступно).
4.  **Экран гонки («Космический Слалом»):**
    *   Анимация полета кораблей (позиции синхронизированы с сервером, но имеют плавную интерполяцию на клиенте).
    *   Визуализация событий: использование бустов, пролет мимо астероидов.
    *   Финишная черта и объявление победителя.
5.  **Результаты:**
    *   Подиум победителей.
    *   Обновление баланса (начисление выигрыша или подтверждение списания).
    *   Кнопка «Играть снова» с теми же параметрами.

#### 3.2. Логика игры (Backend)
*   **Генерация победителя:** Честный алгоритм взвешенной случайности.
    *   Базовый вес игрока = 1.
    *   Вес игрока с бустом = 2 (коэффициент настраиваемый).
    *   Боты имеют базовый вес.
    *   Победитель выбирается пропорционально сумме весов всех участников.
*   **Боты:** Автоматическое заполнение свободных мест ботами по истечении таймера ожидания. При победе бота банк «сгорает» (возвращается системе).
*   **Таймеры:** Строгий контроль времени на подключение, покупку буста и длительность раунда.

#### 3.3. Административный модуль
*   **Конфигуратор комнат:** Создание шаблонов с настройкой:
    *   Стоимости входа и размера банка (% возврата).
    *   Параметров буста (цена, коэффициент влияния).
    *   Лимита участников и времени ожидания.
*   **Валидация экономики:** Предупреждения о некорректных настройках (отрицательная маржа, дисбаланс рисков).
*   **Журнал аудита:** Просмотр детальной истории каждого раунда (кто играл, кто купил буст, расчет весов, итоговый победитель) для подтверждения честности.

### 4. Архитектурные требования

#### 4.1. Структура приложения
*   **Монолитная архитектура (для MVP):** Бэкенд Spring Boot обслуживает API и статику (или работает в паре с отдельным фронтенд-сервером в режиме разработки).
*   **Состояние комнат:** Хранение активных сессий в памяти сервера (In-Memory) с периодической синхронизацией критических данных в БД. Для масштабирования предусмотрена возможность вынесения состояния в Redis.
*   **Frontend Архитектура (React):**
    *   Использование **Context API** для глобального управления состоянием (пользователь, баланс, активная комната).
    *   Кастомные хуки (`useWebSocket`, `useGameLogic`) для инкапсуляции логики подключения и обработки событий.
    *   Компонентный подход: переиспользуемые UI-элементы (кнопки, карточки, модальные окна).

#### 4.2. Real-time взаимодействие
*   Использование протокола STOMP поверх WebSocket.
*   Топики для подписки:
    *   `/topic/rooms` — обновления списка комнат.
    *   `/topic/room/{id}` — события внутри конкретной комнаты (подключение, старт, финиш).
    *   `/queue/user/{id}` — личные уведомления (результаты, ошибки).

### 5. Требования к развертыванию и запуску

Приложение должно быть легко развернуто разработчиками и экспертами на различных платформах.

#### 5.1. Поддерживаемые платформы
*   **Windows 10/11:** Полная поддержка через PowerShell/CMD. Наличие инструкций по установке JDK, Maven, Node.js.
*   **Linux (Ubuntu/CentOS):** Поддержка через терминал, скрипты автоматизации (`start-all.sh`), возможность работы как systemd-сервис.
*   **Docker:** Универсальный способ запуска через `docker-compose up` без необходимости установки локального ПО (кроме Docker Desktop).

#### 5.2. Документация
В репозиторий должен быть включен файл `README.md`, содержащий:
*   Описание стека и архитектуры.
*   Пошаговые инструкции по запуску для **Windows**, **Linux** и **Docker**.
*   Руководство по устранению частых ошибок (порты, переменные окружения).
*   Сценарии тестирования для проверки функционала.

---

### 6. Сценарии приемочного тестирования
Эксперты должны успешно пройти следующие сценарии:
1.  Запуск приложения на локальной машине (выбор ОС: Windows или Linux).
2.  Регистрация/Вход, проверка начального баланса.
3.  Фильтрация комнат, вход в комнату с конкретными параметрами.
4.  Покупка буста, ожидание ботов, наблюдение за стартом гонки.
5.  Визуальная проверка корректности анимации «Космический слалом» (победа соответствует логике).
6.  Проверка обновления баланса после игры.
7.  Вход под администратором: создание новой конфигурации комнаты с предупреждением о рисках.
8.  Проверка журнала раундов на предмет прозрачности результатов.

---

### 7. Экономическое обоснование (для документации)
В проекте должна присутствовать модель расчета эффективности:
*   **Маржа системы:** Формируется за счет комиссии с каждого раунда и случаев победы ботов.
*   **Привлекательность для игрока:** Высокая динамика (раунды по 1 мин), прозрачность шансов (весовая система), возможность влиять на вероятность через буст.

### 8. Ожидаемый результат
1.  **Исходный код:** Репозиторий с разделением на `backend` (Spring Boot) и `frontend` (React + Vite).
2.  **Рабочий прототип:** Возможность запустить полную цепочку «Вход -> Игра -> Выплата» локально.
3.  **Документация:** Актуальный `README.md` с инструкциями для Windows/Linux/Docker и описанием архитектуры.
4.  **Демонстрация:** Готовность к презентации ключевых механик и экономической модели.

> **Примечание:** Визуальный стиль должен строго соответствовать метафоре «Космического слалома», избегая любых элементов, характерных для традиционных азартных игр (карты, рулетка, фишки).
